# 开发底线原则 (Development Principles)

## 单一职责
- 一个模块、一个函数只干一件事
- 如果一个函数需要"和"字来描述它做什么，就该拆成两个
- 避免"万能函数"——改一处影响十处

## 最简代码
- 能删的代码一定删，绝不"留着以后万一有用"
- YAGNI (You Aren't Gonna Need It)
- 没用的代码不是"中性"的——它占用阅读时间、增加维护成本
- 如果某段代码当前不需要，删除它。Git 历史可以找回

## 类型严格
- 每个变量的类型必须明确，不允许模糊处理
- 不用 `any` 类型（TypeScript）
- 不用无类型字典/Map 替代结构化数据
- 函数入参和返回值必须有明确类型

## KISS (Keep It Simple, Stupid)
- 如果需要解释才能看懂，就是太复杂了
- 优先用朴素解法，不要为了"灵活"引入不必要的抽象
- 避免"聪明代码"——作者写的时候得意，别人看不懂

## 文档置信度
- 不确定接口怎么用？停下来查官方文档，不猜
- 涉及支付、加密、认证等关键功能：必须查一手资料
- "我记得好像是..." → 停下来，去查
- 查不到可靠资料 → 写 blocker 报告，等用户提供
